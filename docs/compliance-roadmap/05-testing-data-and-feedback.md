# 05. 测试集、指标与真实审核反馈

## 1. 为什么必须先建测试集

合规检测不是“功能能运行”就算完成。背景、模糊、裁切、抠图边缘等规则都存在阈值和边界样本，没有固定测试集就无法判断每次调整是变好还是变差。

首批规则开发必须与测试样本同步进行。

## 2. 测试数据目录

公开仓库不建议直接提交来源不明的商品图片。建议提交结构、清单和可公开样本，将受版权限制的样本放在私有对象存储。

```text
tests/fixtures/compliance/
├── manifest.jsonl
├── public/
│   ├── pass/
│   ├── background/
│   ├── composition/
│   ├── quality/
│   └── overlay/
└── expected/
    └── reports/
```

对象存储：

```text
compliance-dataset-private/
├── raw/
├── platform-rejected/
├── platform-approved/
└── annotations/
```

## 3. Manifest 格式

每张测试图对应一行：

```json
{
  "id": "amazon-bg-001",
  "path": "public/background/amazon-bg-001.jpg",
  "platform": "amazon",
  "image_role": "main",
  "category": "general",
  "labels": [
    "background_not_white",
    "background_artifact"
  ],
  "expected": {
    "overall_status": "fail",
    "must_fail_rules": [
      "amazon.main.background.white"
    ],
    "must_not_fail_rules": [
      "universal.file.readable"
    ]
  },
  "source_type": "synthetic_test",
  "license": "project-owned"
}
```

## 4. 首批 100 张测试集

最低组成：

| 类型 | 数量 | 说明 |
|---|---:|---|
| 正常通过图 | 40 | 不同商品形态、颜色和材质 |
| 背景问题 | 20 | 非白、渐变、杂物、矩形原背景、阴影 |
| 构图问题 | 15 | 偏移、过大、过小、碰边、裁切 |
| 质量问题 | 15 | 模糊、压缩、过曝、欠曝、白边 |
| 覆盖问题 | 10 | QR、条码、边框 |

正常通过图需要覆盖：

- 白色商品；
- 黑色商品；
- 透明或半透明商品；
- 毛绒和纺织品；
- 光滑瓶罐；
- 细长商品；
- 多孔结构；
- 反光金属；
- 商品包装带正常印刷文字；
- 商品包装本体带条码。

这些样本用于防止算法只适配普通深色实体商品。

## 5. 样本来源

优先级：

1. 团队自行拍摄或制作；
2. 明确允许使用的开放许可数据；
3. 程序合成的确定性违规图；
4. 用户明确授权用于改进的图片；
5. 平台审核失败图片，仅存私有环境并脱敏。

禁止直接将随机电商图片复制进公开仓库。

## 6. 合成测试样本

很多确定性规则可以程序生成测试图：

- 把纯白背景改为指定灰度；
- 在背景添加不同尺寸的色块；
- 将主体移动到边缘；
- 按比例缩小或放大；
- 添加固定宽度边框；
- 添加 QR 码；
- 添加条形码；
- 应用不同程度高斯模糊；
- JPEG 多次压缩；
- 调整曝光；
- 制造 Alpha 白边和孤立碎片。

建议创建：

```text
scripts/generate_compliance_fixtures.py
```

合成样本的优势是问题强度和期望结果可重复。

## 7. 标注方式

每条规则至少需要：

- 明确正样本；
- 明确负样本；
- 边界样本；
- 容易误报的反例；
- 容易漏报的复杂样本。

对可定位问题增加：

- Bounding box；
- Polygon；
- Mask；
- 问题区域类别。

## 8. 核心指标

## 8.1 规则级指标

每条规则统计：

- Precision；
- Recall；
- F1；
- False positive rate；
- False negative rate；
- Unknown rate；
- P50/P95 耗时。

## 8.2 首发验收目标

对于确定性 blocker：

- Recall ≥ 90%；
- 正常样本误报率 < 10%；
- 无法可靠判断的样本应输出 `unknown`，而不是强行通过；
- 每条失败规则能返回指标；
- 可定位规则能返回证据区域。

对于 quality 规则：

- 第一版优先保证少误报；
- 阈值未稳定前不作为整体 fail 的唯一依据。

## 8.3 系统级指标

- 单张确定性分析 P95 ≤ 3 秒；
- 同一图片同一规则版本结果稳定；
- 修复后复检不得直接复制原报告；
- 修复不能引入新的高置信度 blocker；
- 任务异常必须有可理解错误信息。

## 9. 回归测试

新增或修改规则时必须执行：

```text
规则 Schema 校验
→ 检测器单元测试
→ Fixture 回归测试
→ 全规则报告快照对比
→ 性能测试
```

建议测试命令：

```bash
pytest tests/compliance/unit
pytest tests/compliance/fixtures
pytest tests/compliance/performance
```

每次规则集发布保存基准结果：

```text
reports/compliance-benchmarks/
└── amazon-main-2026.07.1.json
```

## 10. 修复效果测试

每个自动修复器至少验证：

1. 对应问题是否被解决；
2. 是否产生新问题；
3. 是否改变商品主体；
4. 是否可重复执行；
5. 输入异常时是否安全失败。

建议比较：

- 修复前后问题列表；
- 主体 Mask IoU；
- 主体感知哈希；
- 主色差异；
- OCR 和 Logo 区域变化；
- 输出格式和尺寸。

## 11. 平台真实审核反馈

产品内增加：

```text
这张图片上传平台后：
- 审核通过
- 审核失败
- 尚未上传
```

失败时收集：

- 平台和站点；
- 商品类目；
- 平台提示原因；
- 截图；
- 用户最终如何解决；
- 是否允许匿名用于改进。

## 12. 反馈分析

按规则版本统计：

- 预测通过但实际失败；
- 预测失败但实际通过；
- 最常见拒绝原因；
- 误报最多的规则；
- 漏报最多的规则；
- 自动修复后的实际通过率变化；
- 不同站点和类目的差异。

## 13. 隐私与数据安全

- 用户图片默认只用于完成处理，不默认进入训练集；
- 用于改进必须获取明确授权；
- 允许用户删除图片、Mask、报告和反馈；
- 日志不记录完整图片内容；
- 平台截图需脱敏订单、店铺和个人信息；
- 私有测试集不得提交公开仓库；
- 明确数据保留期限。

## 14. 发布门禁

以下任一条件不满足，不发布新的规则集为正式版本：

- 来源未记录；
- 没有正负测试样本；
- blocker 误报明显；
- 性能显著退化；
- 修复器改变商品真实性信息；
- 报告无法复现使用的规则版本；
- 旧规则结果无法回滚。
