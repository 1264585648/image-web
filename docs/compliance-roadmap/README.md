# ProductShot AI 电商主图合规能力升级规划

> 文档状态：规划稿 1.0  
> 目标版本：Compliance MVP  
> 首发范围：Amazon 主图 + Google Merchant 通用商品图规则  
> 核心流程：原图预检 → 问题解释 → 选择性修复 → 修复后复检 → 报告与导出

## 1. 为什么要做这次升级

当前项目已经具备商品抠图、白底合成、主体居中、主体缩放、图片增强和多尺寸导出，并提供尺寸、背景、居中、主体占比与清晰度等基础评分。

但现有检测主要发生在系统完成白底、居中和缩放之后，部分结果由生成参数天然保证，无法真实反映：

- 用户原图存在哪些问题；
- 抠图是否成功；
- 修复过程是否改变商品；
- 图片是否符合指定平台、站点、图片角色和类目的规则；
- 哪些问题可以安全自动修复，哪些必须人工确认。

本次升级目标不是继续增加图片模板，而是将系统升级为可信、可解释、可复检的合规处理链路。

## 2. 产品定位

### 当前定位

AI 商品主图标准化生成工具，附带基础质量检查。

### 目标定位

电商主图合规检查与自动修复工具。

系统不承诺平台审核百分之百通过，而是：

1. 按规则检测高风险问题；
2. 展示具体证据、位置、严重级别和置信度；
3. 对确定性问题提供安全修复；
4. 修复后按同一规则重新检测；
5. 通过真实平台审核反馈持续校准。

## 3. 首发范围

首个可商业化版本只聚焦：

- Amazon 主图；
- Google Merchant 通用商品图片；
- 通用文件、构图、背景和质量规则；
- 首批约 20 条确定性规则；
- 白底、居中、占比、尺寸、格式等低风险自动修复；
- 原图与修复图前后对比；
- 单张检测与单张导出。

暂不作为首发阻塞项：

- Temu、TikTok Shop、Walmart、eBay 的完整规则；
- 类目级全覆盖；
- 复杂侵权判断；
- 商品数量与套装语义识别；
- 完全自动去水印、去文字；
- 保证通过平台审核。

## 5. 总体处理链路

```text
上传图片
  ↓
原图预检 Pre-check
  ↓
规则引擎生成问题列表和修复计划
  ↓
用户选择修复项
  ↓
执行低风险修复 Repair
  ↓
修复后复检 Post-check
  ↓
前后对比、仍需确认的问题、下载与反馈
```

## 6. 核心原则

### 6.1 检测与生成分离

生成模板负责画布尺寸、背景、阴影和输出格式；规则引擎负责判断是否符合指定平台要求，两者不能继续混在同一套固定阈值中。

### 6.2 原图和结果图分别检测

必须保存两份独立报告：

- `pre_check_report`：说明原图的问题；
- `post_check_report`：说明修复后还剩什么问题。

### 6.3 Blocker 优先于总分

总分只用于帮助用户理解整体情况。存在高置信度 `blocker` 时，即使总分较高，也必须判定为不通过。

### 6.4 确定性规则优先

尺寸、文件格式、背景像素、边距、碰边、二维码等优先使用可解释算法；多模态模型只用于复杂语义辅助，不能作为所有规则的唯一判定依据。

### 6.5 自动修复必须可控

安全修复可以默认勾选；可能改变商品内容的修复必须由用户确认；真实性、数量、颜色、知识产权和敏感内容问题只报警，不自动改图。

### 6.6 规则必须可追溯

每条规则必须记录：

- 规则 ID；
- 平台、站点、图片角色、类目；
- 生效时间；
- 来源和来源类型；
- 规则版本；
- 检测器和阈值；
- 自动修复方式；
- 严重程度和置信度策略。

## 7. 成功标准

Compliance MVP 达到以下条件才适合正式对外宣传：

- 上传后可先看到原图问题，而不是直接生成后评分；
- 抠图失败或低置信度不会被当作合规；
- 首批 20 条规则均有独立测试样本；
- 所有失败项能说明原因，并尽可能标记问题区域；
- 自动修复后会按相同规则复检；
- 正常图片误报率受控；
- 可以收集平台实际审核结果；
- 产品文案不使用“保证审核通过”。
