# Task 2.3：失败任务一键重试完成记录

## 本次完成范围

- 新增后端接口：

```http
POST /api/tasks/{task_id}/retry
```

- 仅允许 `failed` 状态任务重试。
- 重试不会覆盖原失败任务，而是使用原任务的 `request_json` 创建一个新的 `queued` 任务。
- 新任务会立即进入后台生成流程，继续沿用：

```text
queued → processing → success / failed
```

- 前端历史列表中，失败任务会自动出现“重试”按钮。
- 点击“重试”后：
  - 调用 `/api/tasks/{task_id}/retry`。
  - 接收新的任务 ID。
  - 复用现有轮询逻辑，自动更新结果区和历史列表。

## 后端行为

### 成功重试

原任务：

```text
status=failed
```

调用重试接口后，会创建新任务：

```text
status=queued
progress=0
current_step=已创建任务，等待生成
```

原失败任务保留在历史中，方便查看失败原因。

### 不允许重试的情况

- 任务不存在：返回 `404 任务不存在`
- 任务不是失败状态：返回 `400 只有失败任务可以重试`
- 原任务参数无法读取：返回 `400 原任务参数无法读取，请重新上传图片后生成`
- 原图不存在：返回 `404 原图不存在，请重新上传商品图`
- 模板不存在：返回 `400 模板不存在，请重新选择模板`

## 验收建议

1. 制造一个失败任务，例如删除某个上传源图后触发生成。
2. 打开历史列表，确认失败任务下出现“重试”按钮。
3. 点击“重试”。
4. Network 中确认请求：

```http
POST /api/tasks/{failed_task_id}/retry
```

5. 确认接口返回的是新的任务 ID，状态为 `queued`。
6. 前端开始轮询新任务，并显示进度。
7. 新任务成功后结果区自动展示图片，历史列表中同时保留原失败任务和新成功任务。
